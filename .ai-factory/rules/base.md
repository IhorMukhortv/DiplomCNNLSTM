# Базові правила проекту (Base Rules)

> Автовизначені конвенції та стандарти для розробки проекту. За потреби ви можете редагувати цей файл.

## Угоди про найменування (Naming Conventions)

- **Файли:** `snake_case` (наприклад, `data_preprocessor.py`, `forecast_router.py`)
- **Змінні:** `snake_case` (наприклад, `generation_history`, `predicted_power`)
- **Функції:** `snake_case` (наприклад, `train_cnn_lstm()`, `fetch_weather_forecast()`)
- **Класи:** `PascalCase` (наприклад, `CnnLstmForecaster`, `TimeSeriesScaler`)

## Структура модулів (Module Structure)

- `app/core/` — Бізнес-логіка, математичні моделі прогнозування, конфігурація та шари CNN-LSTM.
- `app/infrastructure/` — З'єднання з базою даних PostgreSQL/TimescaleDB, збереження/завантаження TensorFlow моделей, клієнти API погоди.
- `app/api/` — FastAPI роутери, авторизація, Pydantic схеми запитів та відповідей.
- `tests/` — Модульні (unit) та інтеграційні тести для алгоритмів обробки даних та API.

## Обробка помилок (Error Handling)

- Обробка інфраструктурних помилок (підключення до БД, запити до API погоди) здійснюється в шарі `infrastructure`.
- У шарі `core` викидаються кастомні доменні винятки (наприклад, `ModelNotTrainedError`, `DataPreprocessingError`).
- У шарі `api` ці винятки перехоплюються для повернення користувачеві стандартизованих відповідей з відповідними кодами HTTP (FastAPI `HTTPException`).

## Логування (Logging)

- Використання стандартного модуля `logging` з налаштованими форматизаторами для логування всього життєвого циклу прогнозування.
- Обов'язкове логування етапів навчання моделі (епохи, значення функції втрат на валідації).
- Логування критичних помилок при збої імпорту телеметрії ФЕС або прогнозів погоди.
