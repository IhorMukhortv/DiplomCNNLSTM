# Розділ 4. Архітектура та програмна реалізація інфраструктури системи

## 4.1. Обґрунтування вибору СКБД TimescaleDB для часових рядів та проектування бази даних

Для стабільного функціонування комп'ютерно-інтегрованої системи короткострокового прогнозування генерації фотоелектричних станцій (ФЕС) критично важливим є вибір оптимальної системи керування базами даних (СКБД). Метеорологічні дані та телеметрія вихідної потужності сонячних електростанцій представляють собою часові ряди (time-series data) — послідовності записів, де кожен рядок прив'язаний до точної часової мітки (timestamp). Для таких даних характерні дві ключові особливості:
- Висока інтенсивність операцій запису (ingestion rate) при отриманні похвилинної чи погодинної телеметрії;
- Переважання запитів на читання за останні часові інтервали (наприклад, зчитування останніх 24 годин для розрахунку поточного прогнозу).

У межах розробки архітектури системи було обрано **PostgreSQL з розширенням TimescaleDB**. TimescaleDB поєднує надійність, гнучкість та ACID-гарантії класичної реляційної бази даних PostgreSQL із високою продуктивністю спеціалізованих баз даних часових рядів (NoSQL TSDB).

Основою TimescaleDB є технологія **гіпертаблиць (hypertables)**. З точки зору розробника та SQL-запитів, гіпертаблиця виглядає як звичайна велика таблиця. Проте на фізичному рівні TimescaleDB автоматично розбиває її на секції (chunks) за віссю часу (а за необхідності — і за віссю простору, наприклад, за ідентифікатором станції). Кожен такий часовий сегмент містить дані за певний інтервал часу (наприклад, 1 тиждень або 1 місяць). 

Це секціонування вирішує головну проблему класичних реляційних таблиць — деградацію швидкості запису та пошуку при накопиченні мільйонів рядків через переповнення B-дерева індексів у оперативній пам'яті. Завдяки гіпертаблицям, операції запису та оновлення індексів завжди виконуються у поточному часовому сегменті, який повністю поміщається в оперативну пам'ять (RAM), що забезпечує стабільну швидкість збереження даних незалежно від загального розміру бази.

Схема бази даних розроблена у вигляді таблиці `pv_telemetry`, яка зберігає часові ряди метеорологічних та генераційних параметрів. Структуру та типи даних проектованої таблиці наведено в таблиці 4.1.

**Таблиця 4.1. Специфікація полів таблиці бази даних `pv_telemetry`**

| Назва поля | Тип даних (SQL) | Ключ / Індекс | Опис фізичного параметра |
| :--- | :--- | :---: | :--- |
| **timestamp** | TIMESTAMPTZ | Primary Key / Index | Часова мітка запису в UTC |
| **temperature_2m** | DOUBLE PRECISION | — | Температура повітря на висоті 2 м, °C |
| **relative_humidity_2m** | DOUBLE PRECISION | — | Відносна вологість повітря, % |
| **cloud_cover** | DOUBLE PRECISION | — | Загальна хмарність, % |
| **direct_normal_irradiance** | DOUBLE PRECISION | — | Пряма нормальна інсоляція (DNI), Вт/м² |
| **diffuse_horizontal_irradiance** | DOUBLE PRECISION | — | Дифузна горизонтальна інсоляція (DHI), Вт/м² |
| **global_horizontal_irradiance** | DOUBLE PRECISION | — | Глобальна горизонтальна інсоляція (GHI), Вт/м² |
| **active_power_kw** | DOUBLE PRECISION | — | Фактична вихідна активна потужність ФЕС, кВт |

Перетворення звичайної реляційної таблиці `pv_telemetry` у гіпертаблицю TimescaleDB здійснюється шляхом виконання спеціальної вбудованої SQL-функції після створення структури:

$$\text{SELECT create\_hypertable('pv\_telemetry', 'timestamp');}$$ (4.1)

Це дозволяє задіяти внутрішні механізми TimescaleDB для оптимізації зберігання та швидкого зчитування ретроспективних вікон даних.

## 4.2. Проектування архітектури асинхронного REST API на FastAPI

Для забезпечення взаємодії між базою даних часових рядів, обчислювальним ядром нейромережі CNN-LSTM та зовнішніми споживачами прогнозів (наприклад, веб-інтерфейсом диспетчера чи автоматизованою системою керування енергомережі) розроблено асинхронний REST API.

Основою програмної реалізації сервера обрано сучасний веб-фреймворк **FastAPI**. Головними перевагами FastAPI є:
- **Асинхронність за замовчуванням:** Фреймворк підтримує специфікацію ASGI (Asynchronous Server Gateway Interface) та працює на базі асинхронного циклу подій (event loop) Python `asyncio`. Це дозволяє серверу обробляти тисячі паралельних з'єднань з мінімальними витратами оперативної пам'яті, оскільки під час очікування операцій введення-виведення (I/O), таких як запити до бази даних або мережеві запити, потік виконання не блокується, а перемикається на інші завдання.
- **Швидкість роботи:** FastAPI побудований на базі високопродуктивних бібліотек Starlette (веб-компоненти) та Uvicorn (ASGI-сервер), що наближає його продуктивність до рішень на мовах Go або Node.js.
- **Автоматична валідація та документування:** Завдяки інтеграції з бібліотекою Pydantic, FastAPI виконує сувору типізацію та автоматичну валідацію вхідних JSON-запитів перед виконанням бізнес-логіки. Також автоматично генерується інтерактивна документація API за стандартами OpenAPI (Swagger UI).

Для забезпечення повної асинхронності на всіх рівнях взаємодії з даними, для роботи з базою даних TimescaleDB використовується ORM **SQLAlchemy 2.0** спільно з асинхронним драйвером **asyncpg**. Асинхронний менеджер сесій гарантує, що запити до СКБД не блокуватимуть event loop сервера FastAPI, дозволяючи обробляти інші запити користувачів під час виконання складних SQL-операцій.

## 4.3. Автоматична інтерактивна специфікація інтерфейсів OpenAPI (Swagger UI)

Для стандартизації взаємодії з клієнтськими додатками та полегшення процесу інтеграції розроблений REST API використовує сучасну технологію автоматичного генерування специфікацій за стандартом **OpenAPI (раніше відомим як Swagger)**.

Фреймворк FastAPI при запуску сервера динамічно аналізує опис класів Pydantic (схеми валідації вхідних та вихідних параметрів) та маршрути (endpoints) і генерує стандартизовану схему у форматі JSON за шляхом `/openapi.json`. Ця специфікація містить повний опис усіх підтримуваних HTTP-методологій, шляхів, типів даних, необхідних заголовків та форматів помилок.

На базі цієї OpenAPI-схеми в систему вбудовано два інтерактивні графічні інтерфейси для розробників:
1. **Swagger UI (доступний за шляхом `/docs`):** Надає можливість безпосередньо у браузері переглядати всі доступні маршрути (Health Check `/`, завантаження телеметрії `/api/v1/telemetry/`, отримання прогнозу `/api/v1/predict/`), вивчати схеми запитів/відповідей та виконувати інтерактивні тестові запити ("Try it out") з отриманням реальних відповідей від локального сервера.
2. **ReDoc (доступний за шляхом `/redoc`):** Альтернативний трипанельний інтерфейс документації, адаптований для зручного читання та аналізу структури великих API-сервісів.

Така організація документування дозволяє розробникам та інтеграторам розгортати локальну копію проекту за допомогою Docker Compose або віртуального середовища, запускати сервер розробки і миттєво отримувати повну живу специфікацію з актуальним станом API, без необхідності підтримувати статичні текстові описи в документах, що знижує ймовірність виникнення розбіжностей між кодом та документацією.
