# Розділ 7. Практична реалізація та дослідження донавчання

## 7.1. Структура розробленого програмного забезпечення

Програмне забезпечення комп'ютерно-інтегрованої системи короткострокового прогнозування генерації ФЕС розроблено на мові програмування Python з використанням сучасної модульної архітектури. Відповідно до архітектурних вимог, проект розділено на логічні шари із чітким розмежуванням обов'язків (Separation of Concerns).

Загальна структура каталогів та файлів проекту виглядає наступним чином:

```
app/
├── api/                             # Презентаційний шар (FastAPI)
│   ├── v1/                          # Перша версія API
│   │   ├── endpoints/               # Модулі кінцевих точок API
│   │   │   ├── predict.py           # Розрахунок прогнозів та донавчання
│   │   │   └── telemetry.py         # Запис нової телеметрії
│   │   └── router.py                # Головний маршрутизатор версії v1
│   └── schemas/                     # Схеми валідації Pydantic
│       └── telemetry.py             # Схеми для даних телеметрії
├── core/                            # Шар бізнес-логіки та ML
│   ├── data/                        # Модулі обробки даних
│   │   └── dataset.py               # MinMaxScaler, розбиття та ковзне вікно
│   ├── models/                      # ML-моделі та алгоритми навчання
│   │   ├── cnn_lstm.py              # Архітектура мережі CNN-LSTM
│   │   ├── metrics.py               # Функції розрахунку MAE, RMSE, MAPE
│   │   └── online_learning.py       # Логіка онлайн-донавчання моделей
│   └── config.py                    # Глобальні налаштування (Pydantic Settings)
├── infrastructure/                  # Шар інфраструктури
│   ├── db/                          # Підключення до БД (TimescaleDB)
│   │   ├── models.py                # Опис ORM-моделей SQLAlchemy
│   │   └── session.py               # Асинхронний сешн-факторі та ініціалізація
│   └── storage/                     # Збереження моделей
│       └── model_registry.py        # Потокобезпечний реєстр моделей
└── main.py                          # Точка входу в FastAPI додаток
```

Така структура гарантує високу підтримуваність коду: шар `core` повністю ізольований від деталей роботи бази даних чи HTTP-протоколу, а зміни у презентаційному шарі не впливають на ML-алгоритми.

## 7.2. Опис розробленого REST API та інтеграції реєстру моделей

Взаємодія системи із зовнішніми споживачами (SCADA, диспетчерські пункти) здійснюється через асинхронний REST API. Для реалізації API використано високопродуктивний веб-фреймворк FastAPI, що працює на базі ASGI-сервера Uvicorn.

Усі інтерактивні специфікації та документація до розроблених інтерфейсів генеруються автоматично за стандартом OpenAPI (Swagger UI) і доступні за адресою `/docs` або `/redoc`. Розробник може розгорнути проект локально та інтерактивно перевірити кожен ендпоінт.

Основні розроблені кінцеві точки (Endpoints) включають:

1. `POST /api/v1/telemetry/` — приймає та валідує нові годинні виміри генерації та метеопараметрів від датчиків ФЕС. Дані зберігаються у вигляді гіпертаблиць TimescaleDB для оптимізації роботи з часовими рядами.
2. `GET /api/v1/predict/` — розраховує прогноз вихідної потужності на наступну годину. Запит виконує SQL-вибірку останніх 24 записів, масштабує їх ознаки через `global_scaler` та запускає інференс нейромережі.
3. `POST /api/v1/predict/retrain` — виконує асинхронне мультимасштабне донавчання (fine-tuning) клону базової моделі CNN-LSTM на основі останніх $N$ накопичених у базі даних записів телеметрії та зберігає результат у відповідний іменований слот ансамблю. Приймає параметри: слот ансамблю (`year` або `month`), кількість годин, кількість епох, швидкість навчання та розмір батчу.

Для запобігання конфліктам при паралельних запитах інференсу та донавчання було впроваджено потокобезпечний реєстр `ModelRegistry`. Він використовує об'єкт взаємного виключення (mutex) з бібліотеки `threading` для синхронізації доступу до ваг нейромережі.

## 7.3. Результати симуляційного порівняльного аналізу стратегій ансамблювання

Для оцінки ефективності розробленого мультимасштабного ансамблю було проведено розширений симуляційний експеримент за допомогою скрипту `evaluate_online_learning.py`. Протягом періоду тривалістю в один тиждень (168 годин) з тестової вибірки симулювався реальний режим експлуатації системи, під час якого порівнювалися чотири конфігурації моделі:

1. **Базова модель:** Здійснювала прогнозування на основі статичних ваг, отриманих при первинному навчанні на 3-річному історичному датасеті (без онлайн-адаптації).
2. **Адаптивна модель (кумулятивне Fine-Tuning):** Раз на добу (кожні 24 години) проходила кумулятивну процедуру донавчання на ковзному вікні останніх 168 годин (5 епох, $\eta = 0.0001$). Кожне наступне донавчання продовжувало оптимізацію від попередніх ваг.
3. **Послідовний ансамбль (Базова + 3 версії):** Використовував ансамбль із базової моделі та до 3 останніх послідовних версій кумулятивного донавчання. Прогноз розраховувався як середнє арифметичне.
4. **Мультимасштабний ансамбль (Base + Year + Month):** Запропонований мультимасштабний темпоральний ансамбль, що складається з базової моделі та двох клонів, донавчених на різних горизонтах — річному ($\sim 8760$ годин) та місячному ($\sim 720$ годин). Місячна модель оновлювалася щодоби. Кожне донавчання завжди починалося від свіжого клону базової моделі.

Результати порівняльного аналізу точності прогнозів чотирьох конфігурацій наведено в таблиці 7.1.

**Таблиця 7.1. Результати порівняння стратегій ансамблювання моделей**

| Метрика точності | Базова модель | Адаптивна (FT) | Послідовний ансамбль | Мультимасштабний ансамбль | Од. |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **MAE** | 0.2711 | 0.3939 | 0.3565 | **0.2974** | кВт |
| **RMSE** | 0.4886 | 0.6417 | 0.5863 | **0.5082** | кВт |
| **MAPE** | 9.0623 | 9.9093 | 9.3281 | **8.5396** | % |

Аналіз отриманих числових результатів симуляції демонструє такі закономірності:

1. **Ефективність базової моделі:** Базова модель показує високу точність прогнозування на тестовому тижні з **MAPE = 9.06%** та **MAE = 0.2711 кВт** ($0.9\%$ від встановленої потужності 30 кВт). Це підтверджує стійкість структури CNN-LSTM при роботі зі стандартними сезонними коливаннями.
2. **Проблема кумулятивного донавчання:** Одиночна адаптивна модель з кумулятивним донавчанням показала деградацію якості (**MAPE = 9.91%**, **MAE = 0.3939 кВт**). Це математично пояснюється кумулятивним дрейфом ваг: кожне послідовне донавчання зміщує модель далі від оригінального оптимуму базової моделі, що призводить до локального перенавчання під метеоумови останніх днів.
3. **Обмеження послідовного ансамблю:** Послідовний ансамбль (базова + 3 кумулятивні версії) частково компенсував проблему кумулятивного дрейфу (**MAPE = 9.33%**), але не зміг перевершити базову модель, оскільки всі три донавчені версії мали однаковий горизонт (168 годин) і страждали від подібних деформацій вагового простору.
4. **Перевага мультимасштабного підходу:** Мультимасштабний ансамбль (Base + Year + Month) продемонстрував найкращу точність серед усіх конфігурацій — **MAPE = 8.54%**, що на **5.77%** краще за базову модель. Ключовими перевагами мультимасштабного підходу є:
   - Різні горизонти донавчання (рік vs місяць) дозволяють кожній моделі захоплювати патерни на своєму часовому масштабі — від сезонних трендів до актуальних погодних умов;
   - Клонування від базової моделі при кожному донавчанні повністю виключає кумулятивний дрейф ваг;
   - Усереднення трьох моделей з різними «фокусними відстанями» діє як мультимасштабна регуляризація, що згладжує локальні помилки кожного горизонту.
5. **Аналіз часових діаграм:** Порівняльний графік, збережений у файлі `online_learning_comparison.png`, наочно демонструє, що мультимасштабний ансамбль найточніше відтворює пікові значення генерації у сонячні години. Нижній підграфік абсолютних похибок підтверджує систематичне зниження помилок мультимасштабного ансамблю порівняно з базовою моделлю, особливо у критичні години максимальної генерації.

Таким чином, запропонований мультимасштабний темпоральний ансамбль, натхненний концепцією ковзних середніх з різними періодами (EMA) з фінансового аналізу, успішно вирішує проблему балансу між стабільністю та адаптивністю. Він є єдиною стратегією з досліджених, яка змогла не лише зберегти, а й покращити точність базової моделі, знизивши MAPE з $9.06\%$ до $8.54\%$.
